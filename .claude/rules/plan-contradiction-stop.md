# Rule: Plan contradicts code? STOP and report. Do not work around.

Ralph workers (and other plan-driven implementation agents) receive
scoped beads whose instructions were written by an earlier planning
subagent that may not have re-read the codebase. When a worker
discovers that the plan's stated preconditions are factually wrong,
the correct response is to **STOP, make zero edits, and report the
contradiction** so the orchestrator can reissue with corrected scope.
Do NOT silently reinterpret the plan, do NOT "work around" the
contradiction with best-effort guesses, and do NOT commit partial
work that reflects your reinterpretation.

## The shape of a plan contradiction

A bead says something like "US-005: delete `compare_ab` and the types
it uniquely depends on: `Flip`, `FlipKind`, `ABReport`, `ABArm`".
The worker opens `comparator.py`, searches for `Flip`, and discovers
it is imported and used by `diff_assertion_sets`, which is on the
**preserve** list. The plan's "unique to `compare_ab`" claim is
false. Any attempt to delete `Flip` breaks the preserved function;
any attempt to "work around" this by also deleting
`diff_assertion_sets` silently expands scope beyond the bead and
corrupts the intent of the plan.

## The pattern

```
1. Discover the contradiction (grep, xref, test run, whatever).
2. Make ZERO source edits. Do not stage, do not commit.
3. Report to the orchestrator with:
   - The bead id.
   - The exact plan claim being contradicted (quote it).
   - The evidence (file path, symbol, who else uses it).
   - A suggested corrected scope (optional, but helpful).
4. Exit. Let the orchestrator reissue with a fixed bead.
```

No commit. No partial diff. No "I'll just do the easy half."

## Why this shape

- **Partial work corrupts audit history**: a commit that half-
  implements a bead looks, in `git log`, indistinguishable from one
  that fully implements it. The next worker picking up the epic has
  no way to know scope was silently adjusted mid-flight.
- **The plan is authoritative for intent, not for facts**: the
  orchestrator trusts the plan's intent (what the user asked for)
  but cannot trust every factual assertion inside it. Workers close
  that gap by verifying preconditions and escalating mismatches.
- **Reissue is cheap; rework is expensive**: a reissued bead with
  corrected scope takes the orchestrator seconds. A partial commit
  that later has to be reverted or untangled burns an entire session.
- **Workarounds hide the root cause**: if a worker silently fixes
  the plan's bad precondition, the planning subagent never learns
  to look for this failure mode next time. Escalation is the only
  signal that drives plan-quality improvement.

## Canonical example

#28 US-005 delete list named `Flip` and `FlipKind` as "unique to the
pair-run path". The first worker correctly stopped on discovering
they were shared with `diff_assertion_sets`, reported the
contradiction with a zero-line diff, and the orchestrator reissued
the bead with a corrected delete list. See the commit for
`clauditor-0qb.5` and the plan's `plans/super/28-baseline-pair-delta.md`
Session Notes for the exchange.

## When this rule applies

- Ralph worker beads generated by a planning subagent.
- Any implementation bead whose scope references specific files,
  symbols, or preconditions the bead author has not re-verified.
- Code-rewrite tasks driven from a top-down spec.

## When this rule does NOT apply

- Minor wording issues or typos in the bead text that do not affect
  scope. Those can be silently corrected in the commit message.
- Cases where the contradiction is inside the worker's own freedom
  of movement (e.g. "refactor this module however you like"). If the
  plan did not constrain the approach, there is nothing to contradict.
